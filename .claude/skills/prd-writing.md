# PRD 작성 (Product Requirements Document)

## 스킬 설명
제품 또는 기능 개발에 필요한 요구사항을 PM, 개발, 디자인, QA 등 모든 이해관계자가
공동의 이해를 가질 수 있도록 구조화하는 스킬이다.

## 사용 시점
- 새로운 기능/제품 개발 착수 전
- 개발팀 및 디자이너와 공동 이해를 형성해야 할 때
- 범위를 명확히 정의하고 scope creep을 방지해야 할 때
- 이해관계자 리뷰 및 승인이 필요한 경우

---

## 실행 지침

### Step 1: 배경 및 목적 (Why)
1. **배경**: 이 기능을 만들게 된 비즈니스/사용자 맥락
2. **목적**: 이 기능이 달성해야 하는 목표 (OKR 연결)
3. **성공 지표**: 기능이 성공했다는 것을 판단하는 측정 가능한 기준

### Step 2: 범위 정의 (What)
**In Scope (포함):**
- 이번 버전에서 반드시 구현할 기능

**Out of Scope (제외):**
- 이번 버전에서 의도적으로 제외하는 기능 (이유 포함)

**향후 고려 (Future Consideration):**
- 다음 버전에서 검토할 항목

### Step 3: 사용자 스토리 및 요구사항

**사용자 스토리 형식:**
> "나는 [사용자 유형]으로서, [목표/원하는 것]을 위해 [행동/기능]을 원한다."

각 스토리에 대해:
- **수용 기준 (Acceptance Criteria)**: Given-When-Then 형식
  - Given: 전제 조건
  - When: 행동
  - Then: 기대 결과
- **우선순위**: Must Have / Should Have / Could Have / Won't Have (MoSCoW)

### Step 4: 기술 고려사항
- 알려진 기술적 제약 또는 의존성
- 성능 요구사항 (응답 시간, 처리량 등)
- 보안/개인정보 요구사항
- 외부 시스템 연동 필요 여부

### Step 5: UI/UX 요구사항
- 와이어프레임 또는 디자인 링크 (있는 경우)
- 핵심 UI 상태 (빈 상태, 로딩, 에러, 성공)
- 반응형/접근성 요구사항

### Step 6: 오픈 이슈 및 결정 필요 사항
- 아직 결정되지 않은 사항과 담당자/기한

## 출력 형식 예시

```
# PRD: [기능명]

**작성일:** YYYY-MM-DD
**작성자:** [PM 이름]
**상태:** 초안 / 리뷰 중 / 승인 완료
**버전:** v1.0

---

## 1. 배경 및 목적
[기능을 만드는 이유와 비즈니스 맥락]

## 2. 성공 지표
- [지표 1]: [현재 값] → [목표 값]
- [지표 2]: [현재 값] → [목표 값]

## 3. 범위
**In Scope:**
- [기능 1]
- [기능 2]

**Out of Scope:**
- [제외 기능]: [이유]

## 4. 사용자 스토리

### US-01: [제목]
**스토리:** 나는 [사용자]로서, [목표]를 위해 [기능]을 원한다.
**우선순위:** Must Have

**수용 기준:**
- Given [전제 조건]
- When [행동]
- Then [기대 결과]

## 5. 기술 고려사항
- [제약/의존성]

## 6. 오픈 이슈
| 이슈 | 담당자 | 기한 |
|------|--------|------|
| [이슈] | [담당] | [날짜] |
```

---

## 품질 기준
- [ ] 배경에 Why가 명확히 기술된다
- [ ] Out of Scope가 In Scope만큼 명확히 정의된다
- [ ] 각 사용자 스토리에 Acceptance Criteria가 포함된다
- [ ] 성공 지표가 정량적으로 정의된다
- [ ] 오픈 이슈에 담당자와 기한이 명시된다
