# 유저스토리 작성 (User Story Writing)

## 스킬 설명
개발팀이 사용자 관점에서 기능을 이해하고 구현할 수 있도록
INVEST 원칙에 기반한 유저스토리와 명확한 Acceptance Criteria를 작성하는 스킬이다.

## 사용 시점
- 기능 요구사항을 개발팀에게 전달할 때
- 스프린트 백로그를 구성할 때
- PRD의 요구사항을 실행 가능한 단위로 분해할 때
- QA 테스트 케이스의 기준을 정의할 때

---

## 실행 지침

### Step 1: 에픽 → 스토리 분해
**에픽(Epic)**: 큰 기능 단위 (수 스프린트 소요)
**스토리(Story)**: 1 스프린트 내 완료 가능한 단위

분해 기준:
- 사용자 역할(Role)별 분리
- 행동(Workflow) 단계별 분리
- 성능/보안 등 비기능 요구사항 분리

### Step 2: 유저스토리 기본 형식

**표준 템플릿:**
```
나는 [사용자 역할]로서,
[원하는 목표/기능]을 원한다.
왜냐하면 [가치/이유] 때문이다.
```

**좋은 유저스토리 체크리스트 (INVEST):**
| 원칙 | 의미 | 확인 질문 |
|------|------|---------|
| **I**ndependent | 독립적 | 다른 스토리 없이 단독 구현 가능한가? |
| **N**egotiable | 협상 가능 | 구현 방법이 유연하게 열려 있는가? |
| **V**aluable | 가치 있는 | 사용자 또는 비즈니스에 가치를 주는가? |
| **E**stimable | 추정 가능 | 개발팀이 크기를 예상할 수 있는가? |
| **S**mall | 작은 단위 | 1 스프린트 내 완료 가능한가? |
| **T**estable | 테스트 가능 | 완료 여부를 명확히 판단할 수 있는가? |

### Step 3: Acceptance Criteria 작성

**Given-When-Then 형식 (BDD 기반):**
```
시나리오: [시나리오 이름]
  Given [사전 조건 / 상태]
  When [사용자 행동]
  Then [기대 결과]
```

**작성 원칙:**
- 각 스토리당 3-7개의 시나리오
- 해피 패스(Happy Path) + 엣지 케이스 포함
- 기술 구현 방법이 아닌 동작(Behavior) 기술
- 테스터가 바로 TC로 사용 가능한 수준

### Step 4: 정의 완료 기준 (Definition of Done)
팀 공통으로 적용되는 완료 기준:
- [ ] 코드 리뷰 완료
- [ ] Acceptance Criteria 전항목 통과
- [ ] 단위 테스트 작성 및 통과
- [ ] QA 검증 완료
- [ ] 관련 문서 업데이트

### Step 5: 스토리 포인트 가이드
상대적 크기 추정 (피보나치 수열 기준):
| 포인트 | 의미 | 예시 |
|-------|------|------|
| 1 | 매우 단순 | 텍스트 변경, 색상 수정 |
| 2 | 단순 | 단일 필드 추가 |
| 3 | 보통 | CRUD 한 항목 |
| 5 | 복잡 | 새 화면 + API 연동 |
| 8 | 매우 복잡 | 다중 연동, 비즈니스 로직 |
| 13+ | 에픽으로 분해 권장 | — |

## 출력 형식 예시

```
## 유저스토리: [에픽명]

---

### US-01: [스토리 제목]
**스토리:**
나는 [역할]로서,
[기능]을 원한다.
왜냐하면 [이유] 때문이다.

**우선순위:** Must Have
**스토리 포인트:** 3

**INVEST 검증:**
- Independent: ✅
- Negotiable: ✅
- Valuable: ✅
- Estimable: ✅
- Small: ✅
- Testable: ✅

**Acceptance Criteria:**

시나리오 1: 정상 [행동] 성공
  Given 사용자가 [상태]에 있을 때
  When [행동]을 하면
  Then [기대 결과]가 나타난다

시나리오 2: [엣지 케이스]
  Given [상태]
  When [행동]
  Then [결과]

**제외 사항 (Out of Scope):**
- [이번 스토리에서 다루지 않는 것]
```

---

## 품질 기준
- [ ] 모든 스토리가 INVEST 원칙을 충족한다
- [ ] Acceptance Criteria가 Given-When-Then 형식으로 작성된다
- [ ] 해피 패스와 엣지 케이스가 모두 포함된다
- [ ] 스토리가 기술 구현이 아닌 사용자 관점으로 기술된다
- [ ] 각 스토리에 우선순위와 스토리 포인트가 포함된다
